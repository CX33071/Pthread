数据量很小时，多线程通常比单线程更慢，主要原因在于多线程带来的额外开销已经超过了并行带来的收益
1.上下文切换
上下文切换指的是CPU在不同线程之间切换执行时，需要保存和恢复线程的运行状态
过程：保存当前线程的寄存器、程序计数器等状态
     加载下一个线程的状态
     切换栈空间
这些操作都要时间，通常要几十到几百纳秒
当数据量很小的时候，线程频繁被调度，上下文切换的时间可能比排序时间还多，因此反而更慢
2.Cache Miss
CPU有多级缓存：L1 Cache     L2 Cache    L3 Cache    Main Memory
单线程排序时数据进入CPU cache重复使用，缓存命中率较高
但是多线程的时候：不同线程可能在不同CPU核上运行
               每个核有自己的L1/L2cache
               数据可能在不同cache之间同步
               会出现：线程访问的数据不在本地cache，需要L1 miss->L2 miss->L3 miss->内存
对于小数据，cache miss的代价非常明显
3.线程创建和销毁开销
创建线程不是免费的
创建线程需要：分配线程栈->创建TCB->注册到调度器->系统调用
线程创建：10µs ~ 100µs
小数据：1µs ~ 10µs
线程创建时间>排序时间
总结：当数据规模较小时，多线程排序的线程创建销毁开销、线程调度产生的上下文切换成本以及多核缓存一致性导致的 Cache Miss都会带来额外时间开销，而排序本身计算量较小，导致这些开销超过并行带来的收益，因此多线程排序反而比单线程慢
